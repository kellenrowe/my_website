About this Project

San Fransisco startup Cloverbook provides software that helps event professionals (DJ's, Photographers, etc.) acquire, manage, and interact with their clients.

Brief: In May of 2020 I began working with Cloverbook to address accumulating user feedback and an opportunity for growth in one of the company's key performance indicators (KPIs) - Activation. Cloverbook considers a user activated after they create three events.

Team: Umang Patel, Vivek Sainanee

My Role: User Research, Style Guide Creation, Problem Identification & Solutions Development, Digital Product Design

Tools Used: Adobe XD, Illustrator, VS Code, Google Docs

Before my first meeting with the team I signed up for the free trial so I could walk through the site as a first time user. With a background in wedding photography and live performance, I was curious to see how the site felt to me - a 31 year old, tech-savvy, former event pro.

During my own user journey, I struggled to understand where the site wanted me to go and how it wanted me to get there. I felt unaccomplished, even after creating an event.

User Research

After communicating with the team about my experience, we decided to dive deeper into the initial experiences of our users via two methods; 1) user observation 2) data analysis.


Research Goals

  • Discover path to key drop off points after initial sign-up.
  • Prioritize related aggregate user feedback.

User Observation

We identified 100 non-activated users to observe. All user observations took place via desktop. Because our goal was to understand the path to key drop off points, we cataloged the following information after new users would land on their home page;

  • Which buttons were clicked first on each screen
  • Number of clicks that navigated back/home
  • Which buttons were clicked multiple times
  • Which buttons were not clicked.
  • From which screen drop off occurred.
  • Elapsed time from login to drop off.

Data Analysis

To prioritize user feedback related to activation we used a matrix model to first categorize each submission. After categorizing feedback from over 900 submissions we focused on the green quadrant - procedural inquiries.


With all procedural inquiries isolated, we tallied and ranked each similar submission with the highest number of tallies being ranked number one. The green box below contains the topics users felt they needed help with.

User Behavior

57% of users dropped off within two minutes of first login.
70% within five minutes.

On average, users didn't complete an event creation until their third time on the event creation page.

Nearly all users navigated one click forward and one click back for their first four clicks.

10% of users found the "New" button for document creation.
0 users collaborated with a client, or created a new document.
63% of users dropped off from the Event page.
22% from the Home page, and the remainder from various pages.

Problem Identification & Solutions

- Problem 1 -

Core functionality has low discoverability and multiple tools complete the same task.

Measured user behavior shows initial confusion is the result of having multiple paths to the same destination. As the color-coded image below shows, new users land on a page where two thirds of the clickable options are duplicates.

Additionally three of Cloverbook's key functionalities (Pipeline, Financials, and Lead Capture) are hidden inside dropdown menus and have extremely low discoverability.

- Solution 1 -

Eliminate duplicates and minimize dropdown content.

To make the user's path to event creation less ambiguous we removed all duplicate buttons. We also grouped related tools closer together to allow for an associative understanding of functionality. This process allowed us to bring Pipeline and Financials out into the global nav., increasing discoverability and user understanding of the software's capabilities.

Lastly, we added a notification badge to the Pipeline tool - cueing the user to see that Lead Capture is actively bringing in leads.

- Problem 2 -

Affordance of Workspaces is ambiguous and low discoverability discourages exploration and document creation.

Observed behavior overlaps several procedural inquiries showing new users aren't seeing or understanding the "New" button, the "Workspaces" button, and the relationship between the two. When users do discover the elements, they spend considerable time attempting to comprehend what each affords and how they work together. This consistently leads to tasks being left incomplete and in some cases unnecessarily restarted.

- Solution 2 -

Change the names of Workspaces and New.
Modify element mapping.

By changing the names to Event Documents & New Document, respectively, the conceptual model becomes tangible to the user and the affordance of each is made clear. Moreover, bringing the New Document element into horizontal alignment with the Event Documents element increases discoverability. Lastly, to encourage element exploration, the interaction is changed to mouse-over instead of a click.

- Problem 3 -

Users are confused about meaning and purpose of Event List.

After users create an event, a list is started on their home page. The list is color coded by booking status, however, under the status quo, no key is provided and very little information can be derived from the page. Furthermore, user observation shows many new users [due to a lack of feedback] click back into the event to verify their input saved correctly.

- Solution 3 -

Bring event/contact info, and booking status tool into event list.
Color code booking status tool.

Bringing the event and contact information into the user's event list will provide instant feedback that their input has been saved while simultaneously affording rapid information recall and contact utilization.

Additionally, with the booking status tool now color coded and built into the event list, users are provided with a key and the means to adjust the status from whichever page suits their needs.

- Problem 4 -

Weak signifiers and low discoverability for primary and secondary CTA's in document creation.

Under the status quo, when users create a document, there is document tool bar in the upper right hand section of the page. When users complete the document they are scrolled well below visible range of the toolbar. The result is that users are often unaware of how to share their documents or even that sharing is the next step. In best case scenarios the user scrolls back to the top of the page to share or utilize any tools. Furthermore, the current font color (#DEDEDF) is misleading and signifies that the tools are inaccessable.

- Solution 4 -

Fix toolbar div, change primary CTA to standard Cloverbook button, and change font color of secondary CTA's.

Fixing the toolbar div will enable the user to see and utilize the CTA's at any point in their creation of a document.

Changing the color of secondary CTA's to #323232 (which the style guide dictates) will signify that the tools are available for use.

Lastly, pulling "Share" (the primary CTA) out of the list and converting it to a standard Cloverbook button provides a cue for the user's attention and action.

Conclusion



Meeting Research Goals

Measuring user behavior allowed us to identify pain points leading to drop off and create human centered solutions to mitigate the decrease in user activation. Qualitative analysis of user emails enabled us to understand and address user needs in the aggregate. In these respects this project met the research goals and fullfilled the needs of Cloverbook users.

Challenges

Constraints on budget limited our ability to broaden the methodology of our research. With screen recordings and user feedback being the only tools at our disposal, it was challenging to identify root causes of user confusion with certitude. Quantitative data collection, cross analysis with user feedback, and a highly iterative process were paramount in the success of this collaborative project.

Looking Forward

The solutions from this project are currenlty in development. Looking forward, Continued user observation would allow us to verify that our solutions solved the problems we identified [in the real world].
